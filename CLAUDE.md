# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A generator that produces a canonical "basic dev board" KiCad project for each
Espressif module with a **PCB antenna + native USB** (12 modules across
S2/S3/C3/C5/C6/H2). Each board = a shared pre-built skeleton (USB-C, 3V3 LDO,
BOOT/EN buttons, LEDs) + the module symbol + two break-out pin headers, wired
with net labels, then autorouted to a complete `.kicad_pcb`.

~95% is fully scripted and deterministic. The only judgement step is curating
each module's `board.yaml`. Read `DECISIONS.md` for the *why*; `README.md` for
the operational guide; `docs/AGENT_END_TO_END.md` for the agent task spec.

## Commands

```bash
uv sync                                    # restore Python env (pinned in uv.lock)
uv run python scripts/resolve_library.py   # detect KiCad/libs -> library.json (ONCE per machine, required first)
uv run python scripts/make.py              # THE command: build + route every module
```

`make.py` chains clean → build → route → render → fab:
- `make.py --clean` wipe prior output first · `--render` also 3D montages · `--fab` also Gerber+drill zips · `--all` all stages · `--no-route` build only · `--no-diff` route D+/D- single-ended.

Single module (the `scripts/lib/` primitives, rarely run directly):
```bash
uv run python scripts/lib/extract_pinout.py "<MODULE>"   # symbol -> modules/<m>/pinout.json
uv run python scripts/lib/build_board.py "<MODULE>"      # the generator -> .kicad_sch/pro/pcb
uv run python scripts/lib/route_board.py "<MODULE>"      # autoroute one board
uv run python scripts/lib/validate.py out/<m>/<m>.kicad_sch       # ERC delta + PDF render
uv run python scripts/lib/fab_export.py "<MODULE>"       # Gerbers + drill -> out/<m>/<m>-fab.zip
```

There is no test suite. Validation is the ERC/render gate below; `<MODULE>` is a
dir name under `modules/` (e.g. `ESP32-S3-MINI-1`).

## Architecture

Pipeline is two layers: **orchestrators** in `scripts/` (run these; each loops
over all modules and works standalone) and **primitives** in `scripts/lib/`
(operate on one module). `make.py` is the single front door over the
orchestrators `build_all.py`, `route_all.py`, `render_boards.py`, `clean.py`.

**Source vs output are split into separate trees** so cleaning is `rm -rf out
build` (clean.py) — never a name-matching heuristic. `modules/<M>/` holds ONLY
the curated source (committed); everything generated lands under `out/<M>/` (the
board) or `build/<M>/` (validation artifacts), both git-ignored & disposable.

Per-module data flow:
- `modules/<M>/board.yaml` — **the only hand-authored file.** Holds the decisions
  the generator can't derive (see schema below). SOURCE.
- `modules/<M>/pinout.json` — GENERATED by `extract_pinout.py` from the symbol;
  never hand-edit. Committed as cached SOURCE (curation reads it).
- `out/<M>/<M>.kicad_sch/pro/pcb`, `fp-lib-table`, `*.pretty/`, asset dirs,
  `route_debug/` — GENERATED by `build_board.py`/`route_board.py` (boards copied
  from the baseline). Disposable.
- `build/<M>/` — GENERATED validation artifacts (ERC JSON, PDF). Disposable.

`build_board.py` derives most wiring automatically (power, GND, EN, USB→D±,
GPIO→`GPIOxx` net labels) and places the module + perimeter-split headers. It
also drops a hidden F-silk order-number marker (`SILK_MARKER_TEXT`, default
`"WayWayWay"`) in the clear band under the module, clearance-checked off every
pad — see `place_pcb.silk_marker`.

### Toolchain (decided in DECISIONS.md — don't substitute)
- Python **only** via `uv run python`. Env has kicad-skip, sexpdata, pyyaml, numpy, scipy, shapely.
- Schematic read/write: **kicad-skip** (`from skip import Schematic`) — round-trip-safe on KiCad 10. (kiutils/kipy/pcbnew were rejected.)
- `kicad-cli` is **NOT on PATH** — its absolute path lives in `library.json` (created by `resolve_library.py`).

## Hard rules

- **Never hand-edit a `.kicad_sch` or `.kicad_pcb`** — go through the scripts.
- **Never modify `baseline-left-en/` or `baseline-right-en/`** — these are the two
  input skeletons (EN button on left/right); the generator auto-picks one per
  module by EN-pin side.
- `library.json`, `out/`, `build/`, `.venv/` are regenerated per machine and
  git-ignored. `out/` is fully disposable (`clean.py` wipes it) — keep any manual
  routing experiments OUTSIDE `out/`, not inside it.

## Validation gate

The bar is **no new ERROR-severity ERC violations vs the baseline** — NOT zero
violations. The skeleton ships with ~58 pre-existing violations by design, so
`validate.py` fails only on new *errors*; new *warnings* are expected and fine
(e.g. connecting GPIO0 to the BOOT button always adds one `pin_to_pin` warning
because the EasyEDA button symbol uses `Unspecified` pin types). Always also
**Read the rendered PDF** — the visual check catches what ERC can't (it once
passed a board with unconnected USB pins).

## board.yaml schema (the curation work)

Most wiring is auto-derived; this file only holds:
- **`do_not_break_out`** — pins that must NOT be exposed (internal flash/PSRAM,
  genuinely reserved). Strapping / JTAG / UART-console pins are **still broken
  out** (normal for a dev board) — document in `notes`, don't exclude. USB D+/D-
  are auto-routed and never headered.
- **`strapping` / `input_only`** — SoC GPIO numbers; these only steer the on-board
  LED away from unsafe pins, they don't exclude pins.
- **`builtin_led`** — the safe (I/O-capable, non-strapping, broken-out) GPIO whose
  pad sits nearest the on-board LED. Get the suggestion from
  `build_board.pick_builtin_led()`; build **hard-errors** on an unsafe choice.
- **`boot`** — the download/boot strapping GPIO the BOOT button pulls low: GPIO0 on
  Xtensa (S2/S3), GPIO9 on most RISC-V (C3/C6/H2), GPIO28 on C5.

> Curation is **symbol-specific, not name-based**: e.g. `SPIIO6/7`/`SPIDQS` are
> reserved on the octal S3-WROOM-1 but freely-usable GPIO on the quad S3-MINI-1.
> Check the per-module atomic14 page (`https://www.atomic14.com/esp32/modules/<slug>/`);
> never exclude by name alone.
