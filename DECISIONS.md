# ESP Dev-Board Generator — Decisions & Process

A record of what we're building, the decisions made, the rationale, and how to
re-run the process. Read this first if you're picking the project back up.

## Mission

Create canonical *basic dev boards* for the Espressif module range — one
reference schematic per module. We add an Espressif module symbol plus
break-out pin headers (GPIO + power + GND) to a pre-built skeleton schematic,
and wire everything using **net labels** (no manual routing).

### Module selection criteria (user decision)

Only modules with **a PCB antenna AND native USB** (USB-OTG or USB-Serial-JTAG).
This excludes the classic ESP32 and ESP8266 (UART only) and ESP32-C2/ESP8684
(no native USB). IPEX/U.FL antenna variants are excluded.

Target set (all confirmed present in the installed symbol library):

| SoC | Modules | USB |
|-----|---------|-----|
| ESP32-S2 | S2-MINI-1, S2-WROOM, S2-WROVER | USB-OTG |
| ESP32-S3 | S3-MINI-1, S3-WROOM-1, S3-WROOM-2¹ | OTG + Serial/JTAG |
| ESP32-C3 | C3-MINI-1, C3-WROOM-02 | Serial/JTAG |
| ESP32-C5 | C5-WROOM-1² | Serial/JTAG |
| ESP32-C6 | C6-WROOM-1, C6-MINI-1/U³ | Serial/JTAG |
| ESP32-H2 | H2-MINI-1 | Serial/JTAG |

¹ WROOM-2 uses octal flash/PSRAM → fewer free GPIO (do-not-break-out pins).
² C5-WROOM-1U is the IPEX variant — excluded.
³ One symbol "ESP32-C6-MINI-1/U" covers both PCB and IPEX variants.
ESP32-C61 matches the criteria but has **no symbol in the library yet** — deferred.

## Data sources

- **Module reference / specs**: https://www.atomic14.com/esp32/llms.txt (and the
  per-module/SoC pages it links). Used for editorial decisions (strapping pins,
  USB mapping confirmation, do-not-break-out pins).
- **Symbols / footprints / 3D**: Espressif KiCad libraries
  (https://github.com/espressif/kicad-libraries), installed via KiCad's Plugin &
  Content Manager. Appears as the **`PCM_Espressif`** library; placed symbols use
  `lib_id` `PCM_Espressif:<module-name>`.

## Toolchain decisions

| Concern | Choice | Why |
|---------|--------|-----|
| Python | **`uv`** (user requirement) | `uv init --bare`; `uv run python ...`. |
| Schematic editing | **`kicad-skip` 0.2.5** (`from skip import Schematic`) | Round-trip-safe on KiCad 10 — **proven**: load→write gave a pixel-identical PDF render and a byte-identical ERC report. It preserves unrecognised tokens (only whitespace changes). Emits harmless "can't parsy" warnings on load. |
| Validation | **`kicad-cli`** (ships with KiCad 10.0.3) | Native v10. NOT on PATH — lives at `/Applications/KiCad/KiCad.app/Contents/MacOS/kicad-cli`. |
| Symbol-lib parsing | **`sexpdata`** (kicad-skip dep) | Deterministic read of `Espressif.kicad_sym`. |

Rejected: `kiutils` for *writing* (targets ≈KiCad 7, rewrites whole file from a
strict model → risks dropping v10 tokens); the IPC API `kipy` (needs a running
KiCad, schematic support immature); `pcbnew` Python (PCB only).

## Validation philosophy (user decision: "validate via screenshots")

Two gates, both run by `scripts/lib/validate.py`:

1. **ERC delta vs the baseline.** The skeleton already has ~58 ERC violations by
   design (unconnected labels, undriven power, an `easyeda2kicad` lib not in the
   global table). So the bar is NOT "zero violations" — it's **"resolve some
   baseline violations and introduce no new ERRORS."** The gate is
   **error-severity only**: new *warnings* are acceptable and sometimes
   unavoidable. Concretely, connecting `GPIO0` to the boot button always adds one
   `pin_to_pin` warning, because the EasyEDA button symbol declares `Unspecified`
   pin types — that is a *correct* connection, not a defect. (An earlier,
   stricter "no new violations of any kind" gate sent an agent into an infinite
   loop chasing an impossible exit 0; `validate.py` now fails only on new
   errors and merely reports new warnings.)
2. **PDF render screenshot.** `kicad-cli sch export pdf` → the PDF is read back
   and visually inspected (the Read tool rasterises PDF pages).

Decision: **source and output live in separate trees** — `modules/<M>/` holds
only the curated source (`board.yaml` + cached `pinout.json`), everything
generated goes under `out/<M>/`. So cleaning is `rm -rf out build` with no
name-matching heuristic, and the source tree stays pristine and reviewable.
(Earlier this was one folder per module; mixing generated files with the source
made a clean rebuild fragile.) Data is layered — auto-extracted truth
(`pinout.json`) vs human/curated decisions (`board.yaml`).

## Board-generation approach (user decision)

**Approach A — a deterministic script** builds each board by cloning the
baseline and injecting the module symbol, headers, and net labels from
`board.yaml`. Chosen over an LLM-per-board approach to honour the
"concrete repeatable scripts over burning tokens" principle. The
curated `board.yaml` removes all judgement, leaving pure mechanical placement.
*(An end-to-end agent run was used to discover the mechanics — see below.)*

### Header layout: perimeter-walk split (user decision)

Two break-out headers, laid out as a dev board (one each side of the module).
Allocation is driven by the module's **physical** layout, NOT the schematic
symbol's left/right sides (those are a drawing convention — almost every symbol
is right-heavy). Physical edges come from the **footprint** pad geometry
(`scripts/lib/footprint_edges.py`); pad numbers match symbol pin numbers.

Finding from surveying all 12 modules: form factor does **not** track the
WROOM/MINI naming, and **10 of 12 are 3-edge** (usable pins on left + bottom +
right; the top edge is 100% GND). Only C3-WROOM-02 and C6-WROOM-1 are 2-edge.

Policy: assign each break-out pin to the header on its **physical side** —
**left-edge pins → left header, right-edge pins → right header, bottom-edge pins
to whichever side of the module's horizontal centre they sit on** — ordered along
the perimeter walk (left edge top→bottom, bottom left→right, right edge
bottom→top) so header pins line up with the module pads without crossing. The
shorter header is then **padded with GND pins** so both headers are the same
length (a symmetric board). Degenerates to plain left/right for 2-edge modules.

(An earlier version cut the perimeter ring at the *count* midpoint; when the two
edges had unequal pin counts that pushed a right-edge pin into the left header,
forcing a cross-board route — hence the switch to physical-side assignment.)

Header construction rules (in `build_board.py`):
- **Spread the headers well clear** of the module body and its pin labels
  (`HEADER_OFFSET` from the module centre to each header).
- Conventional dev-board column: **positive rail (`+3V3` left / `+5V` right) at
  the TOP pointing up; GPIO in the middle; `GND` at the BOTTOM pointing down.**
  (KiCad's `power:GND` symbol has its pin at the top and graphic pointing down
  at rot 0; `+3V3`/`+5V` point up at rot 0 — both placed at rot 0.)
- GPIO labels point **outward** (away from the module).
- Excluded pins (`do_not_break_out`) and any other deliberately-unconnected
  signal pin get an explicit **no-connect flag** (else ERC errors).

### Auto-derived vs curated wiring

Most wiring is **auto-derived** from `pinout.json` + footprint, by pin name:
`3V3`→`+3V3`, `GND`→`GND`, `EN`/`CHIP_PU`→`EN`, `GPIO0`→`GPIO0`,
`*USB_D+`→`D+`, `*USB_D-`→`D-`. Each broken-out signal's net label is its
**primary name** (first `/`-separated token) — necessary because some usable
GPIO are function-named (`MTMS`, `U0TXD`, `ADC2_CH6`), not `GPIOxx`.

**Curated, per module, in `board.yaml`** (user decision — explicit, not
name-guessed): the **do-not-break-out** list, chiefly internal flash/PSRAM pins
(`SPIIO6/7`, `SPIDQS`, `SPI_CS1`, …), plus any per-module net overrides.

### Proposed `board.yaml` schema

```yaml
module: ESP32-C3-MINI-1
symbol: PCM_Espressif:ESP32-C3-MINI-1
do_not_break_out: []        # pin numbers or names — flash/PSRAM/reserved
overrides: {}               # pin (number|name) -> net label, to force a net
strapping: [2, 8, 9]        # GPIO numbers sampled at reset (atomic14/datasheet)
input_only: []              # GPIO numbers with no output driver
builtin_led: GPIO7          # on-board LED pin — safe GPIO pad nearest the LED
boot: GPIO9                 # download/boot strapping GPIO the BOOT button pulls low
notes: >                    # block scalar — capture special-role pins here
  Source: <atomic14 module URL>. Strapping/JTAG/UART/USB pins are still broken
  out per dev-board policy and documented here, not excluded.
```

`boot` is the SoC's download/boot strapping pin that the on-board BOOT button
pulls low: **GPIO0** on the Xtensa parts (ESP32 / -S2 / -S3), **GPIO9** on most
RISC-V parts (-C3 / -C6 / -H2), **GPIO28** on -C5 (from the datasheet / atomic14
page). It varies per module, so it can't be a fixed net name; the generator
aliases the skeleton's `BOOT` net onto this pin (same mechanism as the LED).


`strapping`/`input_only` are still broken out like any other pin; they exist so
the generator can route the **on-board LED** (`builtin_led`) to a pin that is
safe to drive — I/O-capable and not sampled at reset. `build_board.py`
`pick_builtin_led()` proposes the safe GPIO whose pad lands **physically closest
to the LED** on the laid-out PCB (shortest trace), and the build **hard-errors**
if `builtin_led` is a strapping /
input-only / non-broken-out pin. Mechanism: keep the pin's `GPIOxx` label and
add a second `BUILTIN_LED` global-label alias on it (the net carries both names;
KiCad canonicalises to `BUILTIN_LED`, matching the skeleton LED's PCB net). This
adds one expected `multiple_net_names` ERC warning.

### Two findings from the end-to-end run (to handle in the generator)

- **Power-symbol orientation**: place `+3V3`/`GND` symbols oriented from the
  pin's edge (top pin → symbol above pointing down, etc.) so they don't overlap
  the module body. (The experiment placed one `+3V3` on top of the symbol.)
- **NC "X" markers**: each module symbol declares many `no_connect`-type pins
  (e.g. C3-MINI-1 has 14, all on the left edge); KiCad always draws them with an
  X. These are inherent to the Espressif symbol, correct to leave unconnected,
  and not a defect.

## What the symbol gives us for free

Each pin in `pinout.json` has: number, name (alt-functions baked in, e.g.
`GPIO19/USB_D+`), electrical type, derived GPIO number, and position/side.
Notably the **USB D+/D- mapping is in the pin names** — not something we must
hand-curate. Still curated (not in the symbol): strapping pins, input-only pins,
and internal flash/PSRAM pins to leave unconnected. Strapping / input-only pins
are captured as the structured `strapping` / `input_only` lists in `board.yaml`
(consumed by the `builtin_led` picker + build-time safety gate), no longer only
as free-text in `notes`.

## Standard net labels (from the baseline skeleton)

The skeleton pre-defines these nets the module must connect to:
`+3V3` (power symbol), `GND` (power symbol + B.Cu copper pour), `+5V`, and global
labels `D+`, `D-`, `EN`, `BOOT`, `BUILTIN_LED`. `BOOT` and `BUILTIN_LED` are
aliased onto the module's boot-strapping GPIO and on-board-LED GPIO respectively,
per `board.yaml` (the GPIO differs by module — see the schema above).

## How to re-run from scratch

```bash
uv sync                                   # restore env (kicad-skip, sexpdata, pyyaml)
uv run python scripts/resolve_library.py  # -> library.json (set ESPRESSIF_3RDPARTY/KICAD_CLI if auto-detect fails)
uv run python scripts/make.py             # THE command: clean -> build_all -> route_all (--render for 3D)
# or stage by stage across all modules:
uv run python scripts/build_all.py        # build + ERC-validate every curated module
uv run python scripts/route_all.py        # autoroute + DRC every generated board
# single module:
uv run python scripts/lib/extract_pinout.py "ESP32-C3-MINI-1"
uv run python scripts/lib/build_board.py "ESP32-C3-MINI-1"
uv run python scripts/lib/validate.py out/ESP32-C3-MINI-1/ESP32-C3-MINI-1.kicad_sch
uv run python scripts/lib/route_board.py "ESP32-C3-MINI-1"
```
